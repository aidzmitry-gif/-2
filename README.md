# Бронирование 2.0

![CI](https://github.com/aidzmitry-gif/-2/actions/workflows/ci.yml/badge.svg)

Система записи к мастерам с одновременным доступом множества клиентов и корректной работой с часовыми поясами. Главная сложность задачи — **не допустить двойной брони одного слота** даже при шквале одновременных запросов. Решается тремя независимыми рубежами защиты, последний из которых — ограничение на уровне самой базы данных.

Стек: **Node.js + TypeScript (Express)**, **PostgreSQL 16** (в Docker), **Prisma** (миграции; ограничение `EXCLUDE` — ручным SQL), интеграционные тесты на **Vitest** против настоящей БД.

## Три рубежа защиты от двойной брони

Бронь создаётся внутри одной транзакции. Параллельный доступ к одному слоту отсекается тремя независимыми механизмами:

1. **Проверка в коде** — внутри транзакции запросом проверяем, что на пересекающийся интервал нет подтверждённой брони. Отсекает абсолютное большинство конфликтов и даёт понятную ошибку `409`.
2. **`SELECT … FOR UPDATE` по строке мастера** — сериализует параллельные транзакции по одному мастеру: пока одна оформляет бронь, остальные ждут. Это закрывает гонку между «проверил» и «вставил».
3. **Ограничение `EXCLUDE` в БД** — последний рубеж на уровне Postgres, который физически не позволяет сохранить два пересекающихся интервала у одного мастера. Срабатывает, даже если код в обход (другой сервис, ручной `INSERT`, баг) попытается записать конфликт.

Ограничение задано вручную (Prisma его не генерирует) во второй миграции:

```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;

ALTER TABLE bookings
  ADD CONSTRAINT bookings_no_overlap
  EXCLUDE USING gist (
    master_id WITH =,
    tstzrange(start_at, end_at, '[)') WITH &&
  )
  WHERE (status = 'CONFIRMED');
```

Интервал — **полуоткрытый** `[)`: бронь 09:00–09:30 и 09:30–10:00 НЕ пересекаются, а 09:15–09:45 с 09:00–09:30 — пересекаются.

## Часовые пояса

В базе хранится **только UTC** (`timestamptz`). У каждого мастера свой пояс; клиент видит слоты в своём.

**Контрольный кейс из задания:** мастер в `Europe/Moscow` (UTC+3) с приёмом в 09:00 — для клиента из `Asia/Yekaterinburg` (UTC+5) это **11:00**.

```
09:00 Москва  →  06:00 UTC (в базе)  →  11:00 Екатеринбург
```

Слоты строятся в локальном времени мастера, переводятся в UTC, и при выдаче подписываются временем мастера и клиента.

## Лист ожидания

Если слот занят, клиент может встать в очередь (`POST /api/bookings/waitlist`). При отмене подтверждённой брони первый в очереди **автоматически** получает запись в той же транзакции, что и отмена, — без гонок. Освобождение и продвижение транслируются в реальном времени.

## Реальное время (SSE)

`GET /api/events` — поток Server-Sent Events. События `booking_created`, `booking_cancelled`, `waitlist_promoted` приходят в открытые вкладки, и список слотов обновляется без перезагрузки.

## Админка

Доступ по паролю из `.env` (`ADMIN_PASSWORD`), передаётся в заголовке `Authorization: Bearer <пароль>`.

- `GET /api/admin/day?masterId=&date=` — записи и блокировки мастера за день;
- `POST /api/admin/cancel` `{ bookingId }` — отмена (с автопродвижением очереди);
- `POST /api/admin/block` `{ masterId, date, startMinute, endMinute, reason }` — ручная блокировка времени (`kind=BLOCK`).

## Тесты и нагрузка

Интеграционные тесты поднимают приложение на случайном порту и работают с настоящей базой; перед каждым тестом БД очищается и засевается заново.

- **Таймзоны** — чистые функции, включая контрольный кейс и полуинтервалы.
- **Конкурентность** — дважды по 50 параллельных запросов: ровно 1 успех (201) и 49 отказов (409).
- **Лист ожидания** — автопродвижение, повторная отмена, отмена без очереди.
- **API** — health, каталог, таймзоны слотов, авторизация админки, блокировка.

`npm run stress` дополнительно проверяет третий рубеж под нагрузкой: 50 запросов на один слот → ровно одна бронь в БД.

## Бонусы

- **Идемпотентность** — заголовок `Idempotency-Key` в `POST /api/bookings`: повтор с тем же ключом вернёт уже созданную бронь (200), а не дубль.
- **Telegram-уведомления** — при заданных `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` шлются уведомления о создании/отмене/продвижении; без токена — тихо отключены.

## Вопросы для защиты

**1. Зачем ограничение `EXCLUDE`, если конфликты уже проверяются в коде и есть блокировка строки?**
Проверка в коде и `FOR UPDATE` защищают только тот код, который ходит через этот путь. `EXCLUDE` — инвариант на уровне данных: его невозможно обойти. Любой другой сервис, ручной `INSERT`, миграция или баг в приложении не смогут создать две пересекающиеся брони у одного мастера. Это последний рубеж, который гарантирует целостность, даже когда всё остальное ошиблось.

**2. Что произойдёт со стресс-тестом, если выключить блокировку строки (`FOR UPDATE`)?**
Тест всё равно покажет ровно одну бронь. При отключённой сериализации (`BOOKING_SERIALIZE=off`) несколько транзакций пройдут проверку в коде одновременно, но при вставке все, кроме одной, упадут на `EXCLUDE` (SQLSTATE `23P01`), и приложение вернёт им `409`. Разница в том, *где* отсекается конфликт: с блокировкой — заранее и без лишней работы БД, без неё — на коммите. А вот если убрать **оба** механизма (и блокировку, и `EXCLUDE`), появятся двойные брони — это и доказывает, что именно ограничение в БД является настоящей гарантией.

**3. Почему в базе хранится UTC, а не локальное время мастера?**
`timestamptz` фиксирует абсолютный момент времени, не зависящий от пояса и перехода на летнее/зимнее время. Если хранить локальное время мастера, ломается сравнение интервалов (а на нём держится `EXCLUDE`), некорректно работает пересечение броней при разнице поясов и возникают неоднозначные/несуществующие моменты в дни перевода часов. UTC в хранилище + перевод в нужный пояс на границе (ввод/вывод) — единственный надёжный способ; пояс мастера и клиента остаётся атрибутом отображения.
